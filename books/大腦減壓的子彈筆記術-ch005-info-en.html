大腦減壓的子彈筆記術 · Chapter 5 "How to Make the Project System Leak-Proof"

Scheduling + Reminders: Start from Preparation, Not Execution

After building the convergence system centered on project master directory notes (10,000 notes → ~20), we must solve the "leakage" problem. This chapter adds two key mechanisms: ① "Scheduling" — placing every task and action at a position in the project flow where you "will definitely see it," eliminating the need to memorize; ② "Reminders" — redefining their essence: not telling you what to do at what time, but telling you "when to start preparing." The shared philosophy: the system automatically pushes what needs to be done right in front of you; you just follow the flow and execute.

01
Scheduling Philosophy
Why Scheduling Is the Foundation of 'Leak Prevention'
The human brain cannot remember the sequence and timing of all actions. The essence of scheduling isn't creating a rigid timetable — it's 'placing every action where you will definitely see it when doing that task.' Like organizing a home — every item has a fixed spot, and any new item knows where to go. The project master directory note IS that 'fixed spot': as you follow the flow, the next action naturally appears in front of you. No memorization, no leakage.

02
Two-Step Scheduling
Sequence + Placement: The Only Two Actions for Scheduling
Step 1: Go beyond creating a project master directory — arrange tasks in the order you'll advance them. The directory order IS your project advancement sequence. Step 2: When any new task or action appears, don't toss it randomly — place it at its correct position in the project master directory. Repeat these two actions, and your project system stays perpetually 'ordered, visible, and leak-proof.'
✅ 有效做法
New task → insert into directory at correct position → follow the flow
❌ 无效做法
New task → jot down mentally or dump into messy to-do list → forgotten or misplaced
03
New Reminder Paradigm
The Essence of Effective Reminders: Preparation Reminders, Not Execution Reminders
Traditional time reminders fail because they tell you 'do X at Y o'clock' — but you may not be free at Y. Effective reminders are 'preparation reminders' — they don't tell you to execute now, but tell you 'a deadline is approaching; start preparing now.' Example: Course on June 1st → set reminder for May 15th — not telling you to teach on May 15th, but to start preparing. You have ample time to arrange actions calmly, rather than being crushed by last-minute urgency.

04
Evernote in Practice
Evernote Reminder List: Auto-Aggregation at Expiry
Evernote's best reminder design isn't pop-up notifications, but the 'Reminder Items List.' When a note's reminder expires, all due notes automatically appear in the reminder list at the top of the note list. This tells you: these tasks need action and preparation NOW. The setup logic is simple: decide 'when should I start preparing' for this task, and set that date as the reminder.

05
Complete Flow Summary
One Thing Daily: Trust the System, Execute What's in Front of You
After the system is fully built, opening Evernote daily shows you not chaotic low-level tasks, but the few focused project master directory notes at the system's top. Don't worry about things outside your view — the system is thoroughly built, relatively leak-free. Even if something gets missed, it must be unimportant. Occasionally, past-scheduled reminders will surface, telling you which project's preparation phase to advance now. You only need to do one thing: open the system daily, trust the system, and execute the few focused projects, tasks, and actions pushed before your eyes.

💡 核心要点
The true art of leak prevention lies not in stronger memory or longer lists, but in the systematic design of 'position' and 'timing.' Place every action where you'll definitely see it in the advancement flow (scheduling), and set 'when to start preparing' reminders for key milestones. The system automatically pushes what needs to be done right in front of you. You don't need to remember anything — you just need to trust the system and act.
