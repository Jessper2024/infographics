01
Algorithm Definition
Five Characteristics: What Qualifies as an Algorithm?
An algorithm is a description of steps to solve a specific problem, represented in a computer as a finite sequence of instructions. An algorithm must have five characteristics: (1) Input — zero or more inputs; (2) Output — at least one output; (3) Finiteness — terminates after finite steps, no infinite loops; (4) Definiteness — every step has a definite meaning, no ambiguity; (5) Feasibility — every step can be completed through a finite number of executions. Only solution steps satisfying all five conditions qualify as a proper algorithm.

Efficiency Comparison
Gauss Summation: Two Algorithms, Worlds Apart
Computing 1+2+…+100: Algorithm 1 uses a loop with 100 additions (O(n)); Algorithm 2 uses Gauss's formula n(n+1)/2 with one multiplication, one addition, one division (O(1)). This is not just about algorithm quality — it is about thinking style. Young Gauss discovered the formula not because he knew "time complexity," but because he possessed abstract thinking — finding patterns and deriving generalizations from concrete problems. The essence of algorithm design is making the computer do less, more intelligently.

sum = n*(n+1)/2 = 100*101/2 = 5050

03
Efficiency Measurement
Measuring Algorithm Efficiency: From Post-Timing to Pre-Analysis
Post-hoc timing: run the program and measure — heavily dependent on hardware, data scale, and compiler optimization; unreliable. Pre-analysis estimation: estimate the algorithm before writing code based on statistical methods. Core insight: A program's runtime depends on algorithm quality and problem input size, ultimately converging on the count of basic operations. Therefore, we ignore all constants and lower-order terms, focusing only on the fastest-growing term as n→∞ — this is the core idea of Big-O.
"The most reliable method to measure runtime is to count the number of basic operations that consume running time. Runtime is proportional to this count." — Cheng Jie
04
Big-O Derivation
Time Complexity: The Three-Step Big-O Derivation Method
Deriving Big-O: ① Replace all additive constants with 1; ② Keep only the highest-order term; ③ Remove the coefficient of the highest-order term. Example: 3n²+2n+1 → O(n²). Constant order (no loop) = O(1), Linear order (one loop) = O(n), Logarithmic order (multiply by 2 each iteration) = O(log n), Quadratic order (two nested loops) = O(n²). The key insight: Big-O cares only about growth rate as n→∞, not specific numbers.
📐 Three-Step Example:
Original count: f(n) = 3n² + 100n + 2000
① Replace constants: 3n² + 100n + 1
② Keep highest order: 3n²
③ Remove coefficient: n²
→ O(n²)
05
Complexity Panorama
Full Ranking of Common Time Complexities and Their Scenarios
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!). O(1) is the ideal for hash lookups; O(log n) appears in binary search; O(n) in linear traversal; O(n log n) in merge/quicksort; O(n²) in bubble/selection sort; O(n³) in triple loops; O(2ⁿ) in subset enumeration; O(n!) in full permutations. There is also space complexity — measuring the storage space required by an algorithm. Trade time for space or vice versa, depending on resource constraints of the specific scenario.

🔑 Key Takeaway
Algorithm quality determines program efficiency. The five characteristics and four design requirements are the standards for evaluating algorithms. The core of algorithm efficiency analysis is Big-O time complexity — by deriving Big-O, we ignore constants and lower-order terms, focusing only on growth trends. Common time complexities range from O(1) to O(n!) across eight levels, with staggering differences at large data scales. A good algorithm = making the computer do less, more intelligently. Learning to analyze time complexity is essential for becoming a great programmer.
